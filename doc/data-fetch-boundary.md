# 数据获取边界说明

本文档用于收口 App Router 迁移后的页面层数据获取边界，作为 `doc/next-step-baseline.md` 中任务 C 的落地说明。

## 当前结论

- 首页与目录页入口已经统一到 `src/app`，仓库内不存在 `src/pages` 页面双入口。
- 目录浏览与文件预览继续保留客户端获取，核心入口是 `src/components/FileListing.tsx` 中的 `useProtectedSWRInfinite`。
- 搜索继续由客户端发起交互式查询，但搜索结果路径补全已收口到服务端 `src/app/api/search/route.ts` + `src/server/onedrive/routes.ts`，客户端不再逐条 `useSWR('/api/item')`。
- 现阶段缓存策略保持保守，目录、搜索、预览相关 Route Handler 继续使用现有 `Cache-Control` / `force-dynamic` 约束，不做激进重写。

## 为什么目录与预览暂不转服务端

- 当前受保护目录口令存放在浏览器本地存储，服务端首屏无法直接读取该口令。
- `FileListing` 同时承载分页、选择下载、批量下载、预览切换、鉴权跳转等强交互行为，现有客户端 SWR 结构更稳定。
- 若现在强行改成服务端首取，需要额外重做受保护目录透传和首屏失败回退逻辑，超出本轮“保守收口”的范围。

## 为什么搜索补全转到服务端

- 搜索弹层本身仍然是客户端交互，保留前端防抖和即时反馈是合理的。
- 结果路径补全依赖 OneDrive API 差异处理，这属于服务端数据整形，不适合继续散落在客户端逐条补请求。
- 路径补全收回服务端后，客户端搜索链路变成单次 `/api/search` 请求，边界更清晰，也减少了搜索结果渲染时的二次请求放大。

## 本轮后续约束

- 新的页面级数据入口优先放在 `src/app` 页面或 Route Handler，不再新增 `src/pages` 页面回退实现。
- 若后续要把目录/预览首屏迁到服务端，需先重新设计受保护目录口令的服务端可见性，再讨论首屏预取和缓存策略。
